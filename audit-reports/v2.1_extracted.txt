Phase 3 — New Feature Integration & Architecture `n`n`n 4.0 Platform foundation `n`n`n 4.0.1 Database — MongoDB (retained as-is) `n`n Per locked decision, no DB migration. Existing MongoDB connection via MONGODB_URI continues. We will, however, add indexes and schema extensions outlined below. `n`n`n New indexes: text index on Trip.title/description/destination; compound { organizerId:1, status:1, createdAt:-1 } on Trip; { targetId:1, rating:-1, createdAt:-1 } on Review; { status:1, priority:-1, slaDeadline:1 } on VerificationRequest; unique { provider:1, eventId:1 } on WebhookEvent; TTL index on WebhookEvent.receivedAt (90d). `n`n`n Connection tuning: retryWrites=true&w=majority&appName=trek-tribe&readPreference=secondaryPreferred. `n`n`n 4.0.2 Caching — Valkey (self-hosted on the same server) `n`n Valkey is a Redis-protocol-compatible OSS fork. It will be deployed on 129.159.22.229 on port 6379 with AUTH and (later) TLS. Because it speaks the Redis wire protocol, ioredis and redis@4 clients already in the project work unchanged. `n`n`n Single env: VALKEY_URL=redis://:<pass>@127.0.0.1:6379 `n`n`n Wrap the existing services/api/src/services/redisService.ts behind a thin abstraction cache/valkeyService.ts so the client can be swapped later. `n`n`n Remove the silent in-memory fallback. When Valkey is unreachable we surface an alert rather than degrading. `n`n`n 4.0.3 Media — pluggable StorageProvider (future-proof) `n`n Instead of hard-wiring to Cloudinary, we introduce a provider interface that defaults to local server disk and can be swapped to Cloudinary or S3 by changing a single env variable. `n`n interface StorageProvider { `n`n upload(buffer: Buffer, meta: UploadMeta): Promise<StoredFile>; `n`n getUrl(key: string, opts?: { ttlSec?: number; download?: boolean }): Promise<string>; `n`n delete(key: string): Promise<void>; `n`n moveTo(destProviderName: string, key: string): Promise<StoredFile>; `n`n } `n`n`n Implementations: LocalDiskProvider (active), CloudinaryProvider (dormant), S3Provider (stub). `n`n`n Env switch: STORAGE_PROVIDER=local | cloudinary | s3. Default = local. `n`n`n Local disk path: /var/trek-tribe/uploads/{type}/{yyyy}/{mm}/{uuid}.{ext} `n`n`n Served by nginx as /uploads/* with immutable caching for content-hashed filenames. `n`n`n All existing upload code (multer, profile photos, KYC docs, trip images, payment proofs, QR) is refactored to go through the StorageProvider. `n`n`n A one-time migration script can copy existing Cloudinary assets down to local (or vice-versa) if the switch is ever flipped. `n `n`n`n 4.1 Authentication, Email & Phone Validation `n`n`n 4.1.1 Firebase Phone Auth (client-driven OTP) `n`n`n Enable Phone, Email/Password, and Google sign-in in Firebase console. `n`n`n Client uses firebase/auth signInWithPhoneNumber — Firebase bears OTP delivery cost (cheap + quota-managed). `n`n`n After sign-in, client calls getIdToken() and sends it as Authorization: Bearer <ID_TOKEN>. `n`n`n Backend middleware firebaseAuth.ts verifies via admin.auth().verifyIdToken(token), upserts User by firebaseUid. `n`n`n RBAC via custom claims: { role, organizerStatus, tier }. requireRole middleware reads claims — no DB hit per request. `n`n`n Dual-run window: keep JWT alongside Firebase for 30 days to avoid lockouts during migration. `n`n`n 4.1.2 Email validation — deep-email-validator `n`n deep-email-validator runs four checks locally without calling external APIs: `n`n`n Regex format validation. `n`n`n Disposable/throwaway domain blacklist. `n`n`n MX record lookup (DNS). `n`n`n SMTP ping — optional RCPT TO probe to confirm mailbox exists. `n`n Integration points: `n`n`n POST /auth/register — reject `n`n`n Any phone field is normalized to +<country><number> before DB write. Stored as normalized string. `n`n`n On registration / profile update: reject invalid numbers with specific reason (too short, wrong country, mobile-only). `n`n`n Firebase Phone Auth receives the normalized E.164 number. `n`n`n For Twilio / WhatsApp dispatch, libphonenumber formats the number according to provider requirements. `n`n`n 4.1.4 User schema additions `n`n { firebaseUid: { type: String, unique: true, sparse: true, index: true }, `n`n authProvider: &quot;password&quot; | &quot;phone&quot; | &quot;google.com&quot;, `n`n phoneE164: String, // normalized `n`n phoneVerifiedAt: Date, `n`n emailVerifiedAt: Date, `n`n emailValidationMeta: { reachable: Boolean, checkedAt: Date, reason: String }, `n`n fcmTokens: [{ token, platform, lastSeenAt }] } `n `n`n`n 4.2 Payments — Razorpay (3% commission, 60/40 escrow split) `n`n`n 4.2.1 Money-flow model `n`n Worked example on a ₹10,000 trip booking: `n`n Step `n`n Amount `n`n Destination `n`n Timing `n`n Customer paid `n`n ₹10,000 `n`n Razorpay capture `n`n At checkout `n`n Platform commission (3%) `n`n ₹300 `n`n Trek Tribe wallet `n`n At capture `n`n Organizer share (97%) `n`n ₹9,700 `n`n — `n`n — `n`n • Advance payout (60% of 9,700) `n`n ₹5,820 `n`n Organizer Razorpay Route account `n`n Within 24h of capture `n`n • Final payout (40% of 9,700) `n`n ₹3,880 `n`n Organizer Razorpay Route account `n`n On trip end date (or T+1) `n`n Refund on pre-trip cancellation `n`n per policy `n`n Customer `n`n Only up to platform wallet balance + unadvanced funds `n`n If an organizer coupon or admin coupon is applied, discount is deducted upstream of this table — see §4.3 Coupon System for split logic. `n`n`n 4.2.2 End-to-end booking flow `n`n`n Client calls POST /api/payments/booking/order with { tripId, seats, packageId, couponCode? }. `n`n`n Server validates seat availability, computes final amount server-side (never trusting client), resolves coupon (§4.3). `n`n`n Server creates razorpay.orders.create({ amount_paise, currency: INR, receipt, notes }). `n`n`n Server persists a Payment row with status=&quot;created&quot; and returns { orderId, keyId, amount, currency, prefill } to client. `n`n`n Client opens Razorpay Checkout. On success, client returns { razorpay_payment_id, razorpay_order_id, razorpay_signature } to POST /api/payments/booking/verify. `n`n`n Server verifies signature via HMAC-SHA256(keySecret, orderId|paymentId) using timingSafeEqual. `n`n`n Server atomically decrements seat capacity, creates GroupBooking, marks Payment authorized, schedules advance-payout job (§4.2.4). `n`n`n NotificationDispatcher fires booking_confirmation across FCM push + email + SMS (Twilio). `n`n`n 4.2.3 Webhook verification pipeline `n`n`n Mount webhook router with express.raw({ type: &quot;application/json&quot; }) BEFORE global express.json(). Raw body is mandatory for signature verification. `n`n`n Signature: expected = hmac_sha256(RAZORPAY_WEBHOOK_SECRET, rawBody). Compare with header x-razorpay-signature using timingSafeEqual. `n`n`n Idempotency: look up eventId via header x-razorpay-event-id in Valkey key wh:rzp:{eventId}. If present → 200 no-op. Else SETEX wh:rzp:{eventId} 172800 1, then insert into WebhookEvent collection. `n`n`n Events handled: payment.authorized, payment.captured, payment.failed, order.paid, refund.processed, payout.processed, transfer.processed, transfer.failed. `n`n`n On payment.captured — schedule advance payout job. On payment.failed — release held seats and notify. `n`n`n On transfer.failed — enqueue retry in RetryJob collection; admin alert after 3 failures. `n`n`n 4.2.4 Payout scheduler `n`n`n Two cron entries in cronScheduler.ts: `n`n`n every-15-minutes — process advance payouts (60% of organizer share) for all Payments captured since last run, provided organizer is &quot;approved&quot; (§4.4). Creates razorpay.transfers under the order. `n`n`n daily at 03:00 IST — process final payouts (40% of organizer share) for bookings whose Trip.endDate < now(). `n`n`n All payouts are recorded in Payout + PayoutLedger collections with idempotency keys. `n`n`n On dispute / chargeback — freeze both advance and final payouts; admin action required to release. `n`n`n 4.2.5 Organizer merchant verification via Razorpay Route `n`n An organizer cannot receive payouts until Razorpay has approved their linked account. The flow is: `n`n`n Organizer fills KYC form in Admin Panel (PAN, business name, phone, bank, GST optional). `n`n`n Server calls razorpay.accounts.create (Route API) with the legal info. `n`n`n Server uploads KYC docs to Razorpay via razorpay.stakeholders endpoints. `n`n`n Server polls Razorpay account status (under_review → activated / needs_clarification / rejected). `n`n`n Admin panel reflects Razorpay status side-by-side with Trek Tribe internal status. Both must be green for payouts. `n`n`n On Razorpay clarification request, organizer is notified and given a to-do list. `n `n`n`n 4.3 Coupon System (Admin + Organizer, split-deduction logic) `n`n`n 4.3.1 Actors & permissions `n`n Actor `n`n Can create `n`n Scope `n`n Funds deducted from `n`n Admin `n`n Platform coupons `n`n Any trip, any category `n`n Primarily platform wallet; optionally share with organizer via per-coupon split `n`n Organizer `n`n Organizer coupons `n`n Only their own trips `n`n Organizer share only — never touches platform commission `n`n`n 4.3.2 Data model — Coupon schema `n`n { code: string (unique, upper-case, 4-20 chars), `n`n createdBy: ObjectId (User), createdByRole: &quot;admin&quot; | &quot;organizer&quot;, `n`n scope: &quot;platform&quot; | &quot;organizer&quot;, `n`n organizerId?: ObjectId, // set only for scope=organizer `n`n tripIds?: [ObjectId], // null = applicable to all of creator's trips `n`n discountType: &quot;percentage&quot; | &quot;flat&quot;, `n`n discountValue: Number, // 0-100 for %, rupees for flat `n`n maxDiscountAmount?: Number, // cap for % coupons `n`n minOrderValue?: Number, `n`n validFrom: Date, validTo: Date, `n`n usageLimitTotal?: Number, usageLimitPerUser?: Number, `n`n usedCount: Number, `n`n // split logic — who bears the discount cost `n`n costSplit: { `n`n platformPercent: Number, // 0-100 `n`n organizerPercent: Number, // 0-100, sum must = 100 `n`n }, `n`n isActive: boolean, `n`n // organizer activation gate (locked decision) `n`n activatedByOrganizer: boolean, // only meaningful for scope=organizer `n`n activatedAt?: Date, `n`n stackable: boolean, // usually false `n`n auditLog: [{ action, byUserId, at, notes }] } `n`n`n 4.3.3 Split logic — who absorbs the discount `n`n Default rules (Admin can override per coupon inside the Admin Panel coupon editor): `n`n Coupon scope `n`n Default split `n`n Override by `n`n Enforcement `n`n Admin / platform `n`n platformPercent=100, organizerPercent=0 `n`n Admin only (per coupon) `n`n Discount amount subtracted from platform commission. If the discount exceeds commission, it eats into platform wallet (never the organizer). `n`n Admin / shared `n`n Admin-configurable e.g. 50/50 `n`n Admin only `n`n Discount split according to costSplit. Organizer must have activatedByOrganizer=true for their share to apply; else entire discount is platform-borne. `n`n Organizer `n`n platformPercent=0, organizerPercent=100 `n`n Organizer cannot change the split `n`n Organizer must activate the coupon (activatedByOrganizer=true) before it can be used. Their 97% share is reduced by the discount. Platform 3% commission is unaffected. `n`n`n 4.3.4 Organizer activation gate (locked requirement) `n`n`n An organizer coupon has a two-stage lifecycle: created → activated. Admin cannot force-activate an organizer coupon. `n`n`n Until activatedByOrganizer=true, the coupon code is rejected at checkout with reason &quot;not activated by organizer&quot;. `n`n`n Activation is performed in the Organizer Dashboard → Coupons tab, with an explicit confirmation: &quot;I understand this discount comes from my 97% share.&quot; `n`n`n Admin can suspend any coupon instantly with an audit-trail reason. `n`n`n 4.3.5 Checkout computation (server-authoritative) `n`n gross = seats * packagePrice; `n`n discount = computeCoupon(gross, coupon); // capped by maxDiscountAmount `n`n customerPays = gross - discount; `n`n platformCommission = gross * 0.03; // commission on GROSS, not net `n`n organizerShareBeforeDiscount = gross - platformCommission; `n`n platformAbsorbs = discount * (coupon.costSplit.platformPercent / 100); `n`n organizerAbsorbs = discount * (coupon.costSplit.organizerPercent / 100); `n`n organizerNetShare = organizerShareBeforeDiscount - organizerAbsorbs; `n`n platformNetIncome = platformCommission - platformAbsorbs; `n`n // Razorpay capture amount = customerPays `n`n // Advance payout (60% of organizerNetShare), Final payout (40%) `n`n`n 4.3.6 Admin Panel UI — Coupon management `n`n`n List view: filters by scope/status/organizer/expiry; columns: code, scope, discount, usage, valid window, activation flag, actions. `n`n`n Create/Edit modal: form with all schema fields; for admin-scope coupons the split sliders (platformPercent / organizerPercent) with live preview of who-pays-what on a sample trip. `n`n`n Audit log drawer: every edit, suspension, restore is recorded with actor + timestamp. `n`n`n Organizer view: only their own trips as target; split fields hidden (organizer-scope always 0/100 to organizer). `n `n`n`n 4.4 Organizer Onboarding — 48-hour SLA manual verification `n`n`n 4.4.1 Status machine `n`n`n User.organizerStatus: &quot;not_requested&quot; → &quot;pending&quot; → &quot;under_review&quot; → &quot;approved&quot; | &quot;rejected&quot; | &quot;suspended&quot;. `n`n`n Side fields: organizerSubmittedAt, organizerReviewedAt, organizerReviewedBy, organizerRejectionReason, organizerApprovalTier (&quot;basic&quot;|&quot;verified&quot;|&quot;premium&quot;). `n`n`n 4.4.2 VerificationRequest schema additions `n`n { ...existing fields, `n`n slaDeadline: Date, // submittedAt + 48h `n`n razorpayAccountId?: string, // Route merchant account `n`n razorpayAccountStatus?: &quot;under_review&quot;|&quot;activated&quot;|&quot;needs_clarification&quot;|&quot;rejected&quot;, `n`n razorpayClarificationNotes?: string, `n`n auditTrail: [{ action, byUserId, at, notes }] } `n`n`n 4.4.3 Funnel & admin workflow `n`n`n Self-register → role=traveler, organizerStatus=not_requested. `n`n`n User clicks &quot;Become an organizer&quot; → POST /api/organizer/apply with business/KYC form. `n`n`n Server creates VerificationRequest{ status:pending }, User.organizerStatus=pending, slaDeadline=now()+48h. `n`n`n User uploads KYC docs (PAN, Aadhaar, business proof, bank statement, GST optional) via signed-URL local storage. `n`n`n When all required docs present → status auto-advances to under_review. `n`n`n Server creates Razorpay Route account asynchronously; Stored on Review at insert time. `n`n F `n`n mean(exp(-age_days / 180)) `n`n Freshness decay — half-weight around 125 days. `n`n Verified `n`n verifiedCount / v `n`n Only post-booking reviews count. `n`n Helpful `n`n sum(helpfulVotes) / (v + 1), clipped to [0,1] `n`n Normalized by platform-wide p95. `n`n`n 4.7.3 Storage `n`n`n Trip.ranking = { score, R_bayes, v, C, updatedAt } with compound index { status:1, &quot;ranking.score&quot;:-1, createdAt:-1 }. `n`n`n Review.sentimentScore : Number in [0,1]; Review.sentiment : &quot;negative&quot;|&quot;neutral&quot;|&quot;positive&quot;. `n`n`n Global constants in Valkey: rank:C (platform mean), rank:v_max (max review count), refreshed every 6h / 24h respectively. `n`n`n 4.7.4 Write-path pipeline `n`n`n On Review create/update/delete (not rejected): aggregate R, v, verifiedCount, helpful sum. `n`n`n Compute F via $exp($divide) in MongoDB aggregation or Node-side. `n`n`n Compute composite S; Trip.updateOne with $set ranking and averageRating/reviewCount. `n`n`n Invalidate Valkey keys: rank:home:*, rank:trip:{id}, rank:organizer:{id}. `n`n`n 4.7.5 Read-path `n`n`n GET /api/recommendations/top?limit=20&region=&category= — sorted by ranking.score desc. `n`n`n Result cached for 10 min in Valkey key rank:home:{region}:{category}:{limit}. `n`n`n Personalization overlay: +10% boost for destinations/categories in User.preferences. Applied post-DB so the base ranking stays explainable. `n`n`n 4.7.6 Anti-gaming `n`n`n Only Review.isVerified===true contributes to R_bayes and Sent. `n`n`n Unique index { reviewerId:1, targetId:1 } prevents multi-reviews. `n`n`n Users with >30% one-star reviews across multiple organizers are down-weighted 0.5x via User.trustScore. `n`n`n Weekly admin digest of suspicious patterns via email + FCM. `n `n`n`n 5.
